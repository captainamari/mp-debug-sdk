## 一、需求摘要

| 维度 | 描述 |
|------|------|
| **核心痛点** | 小程序在真机/体验版/正式版环境缺少像 Chrome DevTools 那样的调试工具，微信内置 vConsole 功能有限（无 Network 面板） |
| **目标用户** | 小程序前端开发者、测试人员 |
| **核心功能** | ① 悬浮按钮（可拖拽） ② Console 日志面板 ③ Network 网络面板 ④ Storage 存储面板 ⑤ 动态远程配置 ⑥ 远程日志上报 |
| **非功能需求** | 零依赖（不依赖 vant 等组件库）、包体积敏感（需压缩）、环境可控（env 控制启停）、不影响业务 |

### v1.1.0 新增需求

| 需求 | 背景 |
|------|------|
| **动态远程配置** | 生产环境不需要所有用户看到调试面板，但紧急情况下需要通过后端配置为某个登录用户（企业ID + 工号）开启调试面板 |
| **远程日志上报** | 为未来监测平台预留，检测异常数据和错误信息；在不影响用户操作的合适时机（页面卸载等）上传数据 |

## 三、设计理念

### 3.1 核心设计哲学

```
低侵入 · 事件驱动 · 优雅降级 · 不影响业务
```

| 原则 | 说明 | 体现 |
|------|------|------|
| **低侵入性** | 最小化对小程序原生 API 的劫持范围 | 仅劫持 `console.*` 和 `wx.request`，不劫持 App/Page/Component 构造器（v1.1.0 中远程日志上报模块需轻量劫持 Page，但只添加 onHide/onUnload 钩子） |
| **事件驱动** | 核心模块与 UI 组件通过事件总线解耦 | 所有数据采集模块通过 `EventBus.emit()` 发布事件，UI 组件通过 `EventBus.on()` 订阅更新 |
| **优雅降级** | 任何异常都不应影响小程序正常运行 | 远程配置超时自动降级、API 劫持失败自动跳过、`try/catch` 保护所有外部调用 |
| **不影响业务** | SDK 不应对用户的业务逻辑产生任何副作用 | 原始 API 总是被调用、远程日志在非关键时机上报、上报请求用独立引用不被拦截 |
| **零依赖** | 不依赖任何第三方库 | 纯原生小程序自定义组件 + 原生 JS 实现 |
| **即插即用** | 最低接入成本 | 复制文件 + 3 行代码即可运行 |

### 3.2 关键设计决策

#### 决策 1：使用 EventBus 而非直接依赖

**问题**：数据采集模块（Console/Network/Error）如何通知 UI 组件更新？

**方案对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| A. UI 组件直接 import 采集模块 | 简单直接 | 耦合度高，扩展困难 |
| B. 发布/订阅（EventBus） | 解耦，易扩展，支持多消费者 | 多了一层间接调用 |
| C. 全局状态管理 | 适合复杂场景 | 小程序原生没有类似 Redux 的方案 |

**选择 B**。EventBus 使得采集模块和 UI 组件完全解耦，同一个事件可以同时被 UI 组件和远程日志上报模块消费。

#### 决策 2：`enabled` 默认值使用 `undefined` 而非 `false`

**问题**：如何判断用户是否显式传入了 `enabled` 参数？

**分析**：SDK 支持两种启用方式：
1. 显式传 `enabled: true/false` → 以 `enabled` 为准
2. 不传 `enabled`，只传 `env` → 根据 env 判断

如果 `enabled` 默认值是 `false`（boolean 类型），`typeof` 判断无法区分"用户传了 false"和"用户没传，使用默认值"。因此默认值使用 `undefined`，通过 `typeof _config.enabled === 'boolean'` 来判断用户是否显式传入。

#### 决策 3：远程配置不阻塞 App() 调用

**问题**：远程配置是异步请求，是否应该等配置返回后再让 App() 执行？

**选择**：**不阻塞**。理由：
- 小程序的启动流程不允许被异步阻塞
- 超时会导致用户感知到启动延迟
- `errorHandler` 必须在 App() 前同步安装（劫持 App 构造函数），因此先安装它，其余拦截器在远程配置返回后再安装

#### 决策 4：远程日志上报使用原始 wx.request 引用

**问题**：远程日志上报的网络请求会被 SDK 自己的 Network 拦截器捕获吗？

**解决**：在安装 Network 拦截器**之前**，保存 `wx.request` 的原始引用到 `wx.__mp_debug_original_request`。远程日志上报模块使用这个原始引用发送请求，避免被拦截。

#### 决策 5：Page 生命周期轻量劫持

**问题**：远程日志上报需要在合适的时机（onHide/onUnload）自动 flush，如何实现？

**方案**：劫持 `Page` 构造函数，为每个新创建的页面注入 onHide/onUnload 钩子。这是一种轻量级劫持——只添加行为，不替换或移除用户原有的回调。

**限制**：只影响劫持之后创建的 Page，已经创建的页面不受影响。定时器 `flushInterval` 作为兜底保证数据最终上报。

## 六、核心模块详解

### 6.1 core/index.js — SDK 入口与初始化

**职责**：SDK 配置管理、初始化流程编排、对外 API 暴露

**初始化流程（两种模式）：**

```
MpDebug.init(options)
         │
         ├── 合并配置
         ├── 保存原始 wx.request 引用
         │
         ├── 有 remoteConfig.configUrl ?
         │   │
         │   ├── 是 → 【异步模式】
         │   │   ├── 1. 同步安装 errorHandler（必须在 App() 前）
         │   │   ├── 2. 标记 _initialized = true
         │   │   ├── 3. 异步请求远程配置
         │   │   │       ├── 返回 enabled:true  → 安装拦截器 + 远程日志
         │   │   │       ├── 返回 enabled:false → 只安装远程日志
         │   │   │       └── 超时/异常/null     → _fallbackToLocalConfig()
         │   │   └── 4. emit 'sdk:ready'
         │   │
         │   └── 否 → 【同步模式】
         │       └── _initWithLocalConfig()
         │           ├── enabled 是 boolean?
         │           │   ├── 是 → 以 enabled 为准
         │           │   └── 否 → 根据 env 判断
         │           ├── 安装 errorHandler + 拦截器 + 远程日志
         │           └── emit 'sdk:ready'
         │
         └── App() 继续执行（不被阻塞）
```

**关键设计点**：
- `enabled` 默认值为 `undefined` 而非 `false`，确保 `typeof` 判断能区分"用户未传"和"用户传了 false"
- `errorHandler` 在异步模式下必须先同步安装（因为需要劫持 `App` 构造函数，而 `App()` 紧跟在 `init()` 后执行）
- `_initialized` 防止重复初始化

### 6.2 event-bus.js — 事件总线

**职责**：提供 publish/subscribe 模式的跨模块通信

**API**：
- `on(event, handler)` — 注册事件监听
- `once(event, handler)` — 注册一次性监听
- `off(event, handler?)` — 移除监听（不传 handler 则移除该事件所有监听）
- `emit(event, ...args)` — 触发事件
- `clear()` — 清除所有事件
- `listenerCount(event)` — 获取监听器数量

**设计要点**：
- 全局单例模式（`module.exports = new EventBus()`），所有模块共享同一个实例
- `emit` 中使用 `try/catch` 保护每个 handler，一个 handler 的异常不影响其他 handler
- `once` 通过 wrapper 函数实现，支持通过 `off` 取消

### 6.3 console-interceptor.js — Console 拦截器

**职责**：劫持 `console.log/info/warn/error/debug`，在保留原始输出的同时采集日志数据

**劫持方式**：
```
原始 console.log → 保存为 originalConsole.log
console.log = 新函数 → 调用原始方法 + 序列化参数 + emit('console:add')
```

**设计要点**：
- 保存原始方法引用到 `originalConsole` 和 `console.__originalLog` 等，供内部日志使用（避免递归）
- `serializeArg()` 安全序列化所有类型的参数，处理循环引用
- `safeStringify()` 使用 `WeakSet` 检测循环引用，限制最大深度
- 日志队列有上限 (`MAX_LOG_COUNT = 500`)，超出时 FIFO 淘汰
- 同时 emit `console:add`（给 UI）和 `log:remote`（给远程上报）

### 6.4 network-interceptor.js — 网络请求拦截器

**职责**：代理 `wx.request`，捕获完整的请求/响应信息

**劫持方式**：
```
原始 wx.request → 保存引用
Object.defineProperty(wx, 'request', { writable: true })
wx.request = 新函数 → 创建记录 + 劫持 success/fail/complete 回调 + 调用原始方法
```

**设计要点**：
- 使用 `Object.defineProperty` 将 `wx.request` 设为可写（小程序默认是只读的）
- 同时劫持 `success`、`fail`、`complete` 三个回调，确保在任何情况下都能记录结果
- `complete` 作为兜底，处理 `success/fail` 未触发的边缘情况
- 请求记录有上限 (`MAX_REQUEST_COUNT = 200`)
- 响应完成时同时 emit `network:response`（给 UI）和 `network:remote`（给远程上报）

### 6.5 error-handler.js — 全局错误捕获

**职责**：劫持 `App` 构造函数，注入 `onError` 和 `onUnhandledRejection` 回调

**劫持方式**：
```
原始 App → 保存引用
App = 新函数(config) → 注入 onError/onUnhandledRejection → 调用原始 App(config)
```

**设计要点**：
- **必须在 `App()` 调用之前安装**，否则劫持无效
- 捕获的错误通过 `consoleInterceptor.addManualLog()` 注入到 Console 面板
- 同时 emit `error:runtime` / `error:promise`（语义事件）和 `error:remote`（远程上报触发）
- `uninstall()` 恢复原始 App 构造函数，但由于 App() 只调用一次，不影响已创建的 App 实例

### 6.6 storage-manager.js — Storage 管理

**职责**：读取和管理小程序本地存储数据

**特点**：
- 纯读取模块，不劫持任何原生 API
- 通过 `wx.getStorageInfoSync()` 和 `wx.getStorageSync()` 直接读取
- 所有操作都有 `try/catch` 保护
- 支持删除单条、清空全部

### 6.7 remote-config.js — 远程动态配置

**职责**：从服务端获取调试面板的启用配置

**请求流程**：
```
fetchConfig(options)
     │
     ├── 无 configUrl → resolve(null)
     │
     ├── 有缓存且未过期? → resolve(缓存)
     │
     └── 发起 GET 请求（带超时控制）
         ├── 200 且数据有效 → 写缓存 → resolve(config)
         ├── 兼容 { code, data } 包装格式
         ├── 非 200 / 数据无效 → resolve(null)
         ├── 超时 → abort 请求 → resolve(null)
         └── 异常 → resolve(null)
```

**设计要点**：
- 所有失败路径都 `resolve(null)` 而非 `reject`，简化调用方的处理逻辑
- 超时使用 `setTimeout` + `requestTask.abort()` 双重保证
- `settled` 标志防止回调重复执行
- 缓存存储在 `wx.Storage` 中，带 TTL 过期机制

### 6.8 remote-logger.js — 远程日志上报

**职责**：收集调试数据并批量上报到远程监测平台

**核心机制**：

```
事件触发 → beforeReport 钩子 → 入队 → 达到阈值/定时/页面卸载 → flush → POST 上报
```

**上报时机**：
1. 队列长度达到 `batchSize`
2. 定时器 `flushInterval` 触发
3. 页面 `onHide`（切后台/跳转）
4. 页面 `onUnload`（页面卸载）
5. `destroy()` 调用（SDK 销毁前 flush）
6. 开发者手动调用 `flush()`

**设计要点**：
- 使用原始 `wx.request` 引用（`wx.__mp_debug_original_request`），避免上报请求被 Network 拦截器捕获
- `beforeReport(type, data)` 钩子允许开发者转换/过滤/脱敏数据，返回 `false` 丢弃
- 上报失败时数据重新放回队列头部，下次 flush 时重试
- 队列有最大长度限制 (`maxQueueSize`)，超出 FIFO 淘汰
- Page 生命周期劫持只影响之后创建的 Page，不影响已存在的页面

## 七、源码阅读指引

### 7.1 推荐阅读顺序

如果你是第一次阅读本项目源码，建议按以下顺序：

```
Step 1: event-bus.js          ← 理解通信机制（最简单的模块）
         │
Step 2: console-interceptor.js ← 理解 API 劫持模式
         │
Step 3: network-interceptor.js ← 同样的劫持模式，更复杂的数据处理
         │
Step 4: error-handler.js      ← 理解 App 构造函数劫持
         │
Step 5: storage-manager.js    ← 最简单的数据模块（无劫持）
         │
Step 6: remote-config.js      ← 理解异步配置拉取和降级
         │
Step 7: remote-logger.js      ← 理解缓冲队列和 Page 劫持
         │
Step 8: core/index.js         ← 最后看入口，理解编排逻辑
         │
Step 9: components/debug-panel ← UI 层如何消费数据
```

### 7.2 每个文件的关键函数

| 文件 | 关键函数 | 作用 |
|------|----------|------|
| `event-bus.js` | `on()`, `emit()`, `off()` | 事件注册/触发/移除 |
| `console-interceptor.js` | `install()`, `serializeArg()`, `addLog()` | 安装劫持、参数序列化、日志入队 |
| `network-interceptor.js` | `install()`, `addRequest()`, `updateRequest()` | 安装劫持、请求记录、响应更新 |
| `error-handler.js` | `install()`, `handleError()` | 安装 App 劫持、错误处理 |
| `storage-manager.js` | `getAllStorage()`, `getStorageInfo()` | 读取 Storage 数据 |
| `remote-config.js` | `fetchConfig()`, `_requestConfig()` | 拉取远程配置、超时处理 |
| `remote-logger.js` | `install()`, `_enqueue()`, `flush()`, `_hookPageLifecycle()` | 安装模块、数据入队、批量上报、Page 劫持 |
| `core/index.js` | `init()`, `_initWithLocalConfig()`, `_fallbackToLocalConfig()`, `destroy()` | 初始化编排、降级处理、销毁 |

### 7.3 阅读时关注的模式

1. **API 劫持模式**：保存原始引用 → 替换为新函数 → 新函数中调用原始 + 额外逻辑
2. **事件发布/订阅**：`eventBus.emit('event', data)` → `eventBus.on('event', handler)`
3. **优雅降级**：`try/catch` 包裹 + 默认返回值 + 条件跳过
4. **缓冲队列**：入队 → 检查阈值 → 批量发送 → 失败重放

## 九、技术风险 & 应对

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| `wx.request` 不可写 | 无法拦截网络请求 | 经验证，`Object.defineProperty(wx, 'request', { writable: true })` 在基础库 2.9.0+ 可行；失败时跳过 Network 拦截 |
| 小程序更新限制 API 重写 | 拦截失效 | 提供降级方案：手动在请求封装层接入 SDK 的 record API |
| Storage 数据量大导致面板卡顿 | 体验差 | 数据截断 + 按需加载 |
| 每个页面需手动引入组件 | 接入稍繁琐 | 提供 app.json 全局注册方式 |
| 跨页面数据不同步 | 新页面看不到旧日志 | 使用 require 模块缓存机制全局存储 + EventBus 跨页面通信 |
| 远程配置超时影响启动 | 用户感知到延迟 | 异步请求不阻塞 App()，超时自动降级为本地配置 |
| Page 劫持与其他框架冲突 | 远程日志上报的 Page 劫持可能与 uni-app 等框架冲突 | 仅在用户显式配置 remoteLogger 时才劫持 Page |
