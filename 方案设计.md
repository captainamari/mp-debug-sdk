> 项目代号：**mp-debug-sdk**

## 一、需求摘要

| 维度 | 描述 |
|------|------|
| **核心痛点** | 小程序在真机/体验版/正式版环境缺少像 Chrome DevTools 那样的调试工具，微信内置 vConsole 功能有限（无 Network 面板） |
| **目标用户** | 小程序前端开发者、测试人员 |
| **核心功能** | ① 悬浮按钮（可拖拽） ② Console 日志面板 ③ Network 网络面板 ④ Storage 存储面板（低优先级） |
| **非功能需求** | 零依赖（不依赖 vant 等组件库）、包体积敏感（需压缩）、环境可控（env 控制启停） |

| **Eruda** | liriliri | H5 调试面板 | Web/H5 | ~150KB | ⭐ 活跃 |

| **uni-devtool** | hx24 | 小程序调试面板 | uni-app | 中等 | ⚠️ 维护较少 |

### 2.2 详细对比

#### 微信内置 vConsole
- **优点**：无需引入，官方支持
- **缺点**：
  - ❌ **没有 Network 面板**（最大痛点）
  - ❌ 功能非常有限，仅支持 console 日志查看
  - ❌ 不支持查看请求头、请求体、响应数据
  - ❌ 不支持 Storage 可视化管理

#### WeConsole（微盟团队）
- **优点**：
  - ✅ 功能最全面，对标 Chrome DevTools
  - ✅ 支持 Console / Api / Component 组件树 / Storage / 自定义扩展
  - ✅ 原生小程序自定义组件，不依赖框架
  - ✅ 支持通过 `replace(scope)` 按需替换
- **缺点**：
  - ❌ 包体积较大（全量引入）
  - ❌ 维护频率不高，社区活跃度一般
  - ❌ 组件树功能过重，对简单场景来说功能过剩
  - ❌ 拦截了 App/Page/Component 构造器，侵入性较强

#### uni-devtool
- **优点**：
  - ✅ 技术方案成熟（Webpack Loader 自动注入）
  - ✅ Network 面板功能完善
- **缺点**：
  - ❌ **强依赖 uni-app 框架**，原生小程序无法使用
  - ❌ 依赖 Vue Event Bus，不适用非 Vue 项目

#### Web 端 vConsole / Eruda
- **定位**：H5 调试工具，通过操作 DOM 实现
- **与小程序的差异**：小程序没有 DOM，不能直接使用，但架构设计思路可借鉴

## 三、最佳实践方案

### 3.1 设计原则

经过竞品分析，我们的方案需遵循以下原则：

| 原则 | 说明 |
|------|------|
| **零依赖** | 不依赖 vant、uni-app 或任何第三方组件库，纯原生小程序自定义组件实现 |
| **轻量化** | 控制包体积，提供 gulp/rollup 压缩构建，目标压缩后 < 50KB |
| **低侵入** | 只拦截必要的 API（console / wx.request），不拦截 App/Page/Component 构造器 |
| **环境可控** | 通过 env 参数控制，默认不启用，显式配置后才激活 |
| **即插即用** | 用户只需复制文件 + 两行代码即可接入 |

### 3.3 技术方案详解

#### 3.3.1 Console 拦截

**思路**：重写 `console.log/info/warn/error` 方法，在执行原始方法的同时，将日志数据推送到内部队列。

```javascript
// 伪代码示意
const originalConsole = {};
['log', 'info', 'warn', 'error'].forEach(method => {
  originalConsole[method] = console[method];
  console[method] = function (...args) {
    // 1. 调用原始方法（保证开发工具中正常输出）
    originalConsole[method].apply(console, args);
    // 2. 格式化并存储日志
    EventBus.emit('console:add', {
      type: method,
      data: args,
      time: Date.now()
    });
  };
});
```

#### 3.3.2 Network 请求拦截

**思路**：使用 `Object.defineProperty` 将 `wx.request` 设为可写，替换为自定义函数，通过劫持 `success/fail/complete` 回调捕获响应。

```javascript
// 伪代码示意
const originalRequest = wx.request;
Object.defineProperty(wx, 'request', { writable: true });

wx.request = function (options) {
  const requestId = generateId();
  const startTime = Date.now();

  // 记录请求信息
  EventBus.emit('network:request', {
    id: requestId,
    url: options.url,
    method: options.method || 'GET',
    header: options.header,
    data: options.data,
    startTime
  });

  // 劫持回调
  const originalComplete = options.complete;
  options.complete = function (res) {
    EventBus.emit('network:response', {
      id: requestId,
      statusCode: res.statusCode,
      header: res.header,
      data: res.data,
      duration: Date.now() - startTime
    });
    originalComplete && originalComplete.call(this, res);
  };

  return originalRequest.call(this, options);
};
```

同时也需拦截 `wx.uploadFile` 和 `wx.downloadFile`。

#### 3.3.3 全局错误捕获

**思路**：利用小程序 App 生命周期的 `onError` 和 `onUnhandledRejection`。

```javascript
// 在 SDK 初始化时，劫持 App 的 onError
const originalApp = App;
App = function (config) {
  const originalOnError = config.onError;
  config.onError = function (err) {
    EventBus.emit('console:add', {
      type: 'error',
      data: ['[Runtime Error]', err],
      time: Date.now()
    });
    originalOnError && originalOnError.call(this, err);
  };

  const originalOnUnhandledRejection = config.onUnhandledRejection;
  config.onUnhandledRejection = function (res) {
    EventBus.emit('console:add', {
      type: 'error',
      data: ['[Unhandled Rejection]', res.reason],
      time: Date.now()
    });
    originalOnUnhandledRejection && originalOnUnhandledRejection.call(this, res);
  };

  return originalApp(config);
};
```

#### 3.3.4 悬浮按钮（可拖拽）

**思路**：使用小程序的 `movable-area` + `movable-view` 组件实现拖拽（官方组件，性能最好），或使用 `catchtouchmove` 事件手动计算。

> **推荐方案**：`movable-area` + `movable-view`，原因：
> - 官方组件，兼容性好
> - 自带惯性滑动和边界处理
> - 代码量最少

```xml
<!-- 悬浮按钮 -->
<movable-area class="mp-debug-float-area">
  <movable-view
    class="mp-debug-float-btn"
    direction="all"
    x="{{btnX}}"
    y="{{btnY}}"
    bindtap="onTogglePanel"
  >
    <text class="mp-debug-float-text">Debug</text>
  </movable-view>
</movable-area>
```

#### 3.3.5 Storage 面板（低优先级）

**思路**：直接调用 `wx.getStorageInfoSync()` 和 `wx.getStorageSync()` 读取所有存储数据，以列表形式展示。

```javascript
function getAllStorage() {
  const { keys } = wx.getStorageInfoSync();
  return keys.map(key => ({
    key,
    value: wx.getStorageSync(key),
    size: JSON.stringify(wx.getStorageSync(key)).length
  }));
}
```

### 3.4 用户接入方式

#### Step 1：复制 SDK 到项目

```
将 dist/mp-debug-sdk/ 复制到项目的 lib/ 目录下
```

#### Step 2：初始化 SDK（app.js）

```javascript
// app.js
import MpDebug from './lib/mp-debug-sdk/index';

// 初始化（需在 App() 调用之前）

  env: 'development',  // 'development' | 'production'
  // 当 env 为 'production' 时不启用，不传则默认不启用
  // 也可以通过自定义逻辑控制
  enabled: __wxConfig.envVersion !== 'release'  // 正式版不启用
});

App({
  // ...
});
```

```json
// page.json

```

### 3.5 包体积优化方案

| 策略 | 说明 | 预估收益 |
|------|------|----------|
| **gulp 构建流水线** | JS 压缩（terser）+ WXML 压缩 + WXSS 压缩（cssnano） | 减少 50-60% |
| **按需引入** | 用户可只引入 Console 模块，不引入 Network/Storage | 减少 30-50% |

| **CSS 内联 & 合并** | 减少文件数量 | 减少文件 I/O |
| **压缩后目标体积** | 全量 < 50KB，核心模块 < 30KB | - |

### 4.2 各面板设计

#### Console 面板
- 日志条目：`[时间] [类型图标] 日志内容`
- 类型颜色：log(白) / info(蓝) / warn(黄) / error(红)
- 顶部过滤：All | Log | Info | Warn | Error
- 对象/数组类型支持展开折叠（JSON 树）
- 底部操作：Clear（清空）

#### Network 面板
- 请求列表：`[方法] URL [状态码] [耗时]`
- 状态码颜色：2xx(绿) / 3xx(蓝) / 4xx(黄) / 5xx(红)
- 点击展开请求详情：
  - **General**：URL / Method / Status / Duration
  - **Request Headers**：请求头 Key-Value
  - **Request Body**：请求体（格式化 JSON）
  - **Response**：返回报文（格式化 JSON）
- 底部操作：Clear（清空）

#### Storage 面板
- Key-Value 列表
- 点击查看详情（格式化 JSON）
- 支持刷新

## 七、技术风险 & 应对

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| `wx.request` 不可写 | 无法拦截网络请求 | 经过验证，`Object.defineProperty(wx, 'request', { writable: true })` 在当前基础库版本可行 |
| 小程序更新限制 API 重写 | 拦截失效 | 提供降级方案：手动在请求封装层接入 SDK 的 record API |
| Storage 数据量大导致面板卡顿 | 体验差 | 分页加载 + 虚拟列表 + 数据截断 |
| 每个页面需手动引入组件 | 接入稍繁琐 | 提供 app.json 全局注册方式 + 文档说明 |
| 跨页面数据不同步 | 新页面看不到旧日志 | 使用 getApp() 全局存储 + EventBus 跨页面通信 |
